**Draft status:** mock-first / source-review. This draft does not use real NASA POWER data yet and is not a public evidence note.

Weather is one of the simplest ways public data becomes personal. A hot stretch, a dry run, or a sudden wet day can shape work, travel, health, household routines, and city life before it becomes part of a policy conversation.

This draft tests one small question:

**What can one month of daily temperature and rainfall context tell us about Delhi?**

The answer has to stay modest. A one-month view can help readers practise seeing daily variation: warmer spells, cooler intervals, dry stretches, and rainfall days. It cannot prove climate change, represent every neighbourhood, replace local station observations, or become a live weather product.

That is the point of the exercise. Track & Tote should be able to show a chart without asking the chart to carry a conclusion it cannot support.

## The working idea

A reviewed weather snapshot can make a short data essay useful when the chart keeps its source, time window, status, and caveats visible.

For now, this note is only testing the essay structure. The values below are placeholders. They should not be read as observed Delhi weather, NASA POWER output, or evidence for any final claim.

## Why a one-month note can still be useful

One month is too small for a climate trend. It is still useful as a reading exercise.

It can show how a public-data note should work before the stakes get bigger:

- keep the question narrow
- show the time window
- keep units and source status visible
- separate observation from interpretation
- place caveats next to the chart

If this structure works for a small weather note, the same discipline can later help with more complex public-data essays.

## How the future snapshot would work

The future workflow should be deliberately slow:

```text
official NASA POWER source
-> source and legal review
-> raw snapshot storage, if permitted
-> normalization
-> reviewed JSON or CSV
-> metadata file
-> note chart component
-> source note and caveat
-> published Note
```

Every snapshot-backed chart should show:

- source
- dataset or version
- retrieved or downloaded date
- latest observation
- update frequency
- data status
- licence status
- confidence
- caveat

If a field is unresolved, the chart should say `needs verification` rather than hiding the gap.

## How to read the future chart

If a reviewed snapshot is later added, the interpretation should stay close to the data.

Safer language:

- the selected month shows daily variation in the reviewed snapshot
- the selected point has warmer and cooler days within the chosen window
- rainfall appears on some days and not others within the chosen window
- this is historical context, not a live feed

Unsafe language:

- Delhi is getting hotter
- rainfall is changing
- this represents all of Delhi
- this proves local exposure or risk
- this is production-approved public data

## What this draft is testing

This draft is testing four things.

First, whether the question is small enough. A month of weather context is easier to explain than a large climate claim.

Second, whether the visual has a clear job. The future chart should help readers see daily variation, not decorate the page.

Third, whether the caveat can sit close to the claim. Readers should not have to search for the limits of the data.

Fourth, whether Track & Tote can publish data-backed Notes without pretending that every dataset is live, complete, or production approved.

## What this note should not say

This note should not say that Delhi is getting hotter based on one month. It should not say that rainfall patterns have changed based on one small window. It should not imply that NASA POWER proves local station conditions. It should not treat a reviewed snapshot as live weather or approved reporting data.

The stronger article is the more careful one.
