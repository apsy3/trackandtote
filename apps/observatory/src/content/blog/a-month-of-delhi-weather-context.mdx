**Draft status:** mock-first planning note. This draft does not use real NASA POWER data yet.

Weather is one of the ways public data becomes personal. A hot stretch, a dry run, or a sudden wet day can shape work, travel, health, household routines, and city life before it becomes part of a policy conversation.

This draft tests a small question:

**What can one month of daily temperature and rainfall context tell us about Delhi?**

The answer has to stay modest. A one-month view may help us notice daily variation, warmer spells, dry stretches, and rainfall days. It cannot prove climate change, represent every neighbourhood, replace local station observations, or become a live weather product.

## The working idea

A reviewed weather snapshot can make a short data essay more useful if the chart keeps its source, time window, status, and caveats visible.

For now, this note is only testing the essay structure. The values below are placeholders. They should not be read as observed Delhi weather, NASA POWER output, or evidence for any final claim.

## Source note for the future snapshot

The planned source is NASA POWER. Track & Tote has not approved NASA POWER for production public charts yet.

Before any real values appear here, the source review must confirm licence, attribution, citation, caching, redistribution, public display, parameter definitions, units, time standard, and caveat language.

The future chart must also explain that NASA POWER values are gridded weather or climate context. They are not local station measurements and should not be treated as exact neighbourhood-level weather.

## How the future snapshot would work

The future workflow should be:

```text
official NASA POWER source
-> source and legal review
-> raw snapshot storage, if permitted
-> normalization
-> reviewed JSON or CSV
-> metadata file
-> note chart component
-> source note and caveat
-> published Note
```

Every snapshot-backed chart should show:

- source
- dataset or version
- retrieved or downloaded date
- latest observation
- update frequency
- data status
- licence status
- confidence
- caveat

If a field is unresolved, the chart should say `needs verification` rather than hiding the gap.

## What this draft is testing

This draft is testing four things.

First, whether the question is small enough. A month of weather context is easier to explain than a large climate claim.

Second, whether the visual has a clear job. The future chart should help readers see daily variation, not decorate the page.

Third, whether the caveat can sit close to the claim. Readers should not have to search for the limits of the data.

Fourth, whether Track & Tote can publish data-backed Notes without pretending that every dataset is live, complete, or production approved.

## What this note should not say

This note should not say that Delhi is getting hotter based on one month. It should not say that rainfall patterns have changed based on one small window. It should not imply that NASA POWER proves local station conditions. It should not treat a reviewed snapshot as live weather.

The stronger article is the more careful one.
